## What this project is

**Argus** is a standalone, self-hosted face + object recognition app. Single Docker container, FastAPI backend, Jinja2 server-rendered UI (no separate frontend framework), SQLite storage. Runs on native Windows via Docker Desktop — **not WSL2**, so do not assume WSL2-specific networking/paths apply here.

Not coupled to any other project (e.g. Shashin). No shared code, no shared containers.

**This must be a generic, standalone tool — not a Shashin feature.** Shashin is the first intended consumer of the API, but treat it exactly like any other hypothetical client: it talks to this app only over HTTP with an API key, same as a curl script or any other future integration would. Concretely:
- Never import Shashin code, never depend on a Shashin library
- Never share a database, filesystem, or container with Shashin
- Never add a route, field, or setting modeled on Shashin's specific concepts (its photo schema, its galleries, its tagging conventions) — only add things that make sense for the generic face/object recognition domain on their own merits
- This app must run and be fully useful standalone, with zero awareness of whether Shashin exists

If something Shashin-specific comes up later, solve it from Shashin's side (Shashin calls the existing/generic API, possibly in a batch) or by adding a genuinely generic capability any client could use — not a Shashin-aware code path here.

## Core stack — do not deviate without discussion

- FastAPI + uvicorn
- Jinja2 templates, vanilla JS/CSS (no React/Vue) — keep frontend deps at zero beyond what ships in browsers
- InsightFace for faces (RetinaFace detection + ArcFace embeddings), used directly — not via CompreFace
- Ultralytics YOLO for objects
- `onnxruntime` AND `onnxruntime-gpu` both available; GPU usage is a real runtime setting (`system.use_gpu`, default `false`), not a build-time CPU-only decision. See "GPU support" in `DESIGN.md` §2.
- Raw `sqlite3` (stdlib) — no SQLAlchemy, no ORM
- numpy cosine similarity for face matching — no faiss (revisit only past tens of thousands of enrolled faces)
- No Celery, no Redis, no job queue framework — use `BackgroundTasks` or asyncio for model downloads
- `httpx` for server-side `image_url` fetching, `pillow-heif` for HEIC/HEIF support — these are the only two deps added beyond the original minimal list, both directly required by the file/URL + multi-format input requirement below

**Minimal dependencies is a hard project value, not a soft preference.** Before adding any new dependency, ask whether stdlib or a few lines of code covers it.

## Run modes — Docker and CLI, one codebase

Argus must run two ways: `docker compose up` (primary) and directly on the host via `python -m app` (no Docker). **Both call the exact same `app/main.py` FastAPI app object** — `app/__main__.py` and the Docker image's entrypoint are both thin `uvicorn.run(app, ...)` wrappers, nothing more. Never let route/business logic differ between the two, and never branch config (API key, host, port, model/data paths) on "am I in Docker or not" — both read the same environment variables / `.env` the same way. Use stdlib `argparse` for any CLI flags before reaching for `click`/`typer`, per the minimal-deps value. Full detail in `DESIGN.md` §2 ("Run modes"). This is separate from the native OS installers (Windows/macOS/Linux) described in the Platform Distribution Roadmap below — `python -m app` is available from day one; packaged installers are a distinct, later phase.

## Versioning & CI/CD

Project starts at **`0.1.0`** (alpha — use `0.1.0-alpha.1` style pre-release tags while in alpha). Full scheme in `DESIGN.md` §3. Key points:

- Single source of truth: `app/__init__.py` → `__version__ = "0.1.0"`. Everything else (health endpoint, Docker image tag) reads from this — never duplicate the version string by hand elsewhere.
- Two GitHub Actions workflows: `test.yml` (lint + pytest + docker build sanity check, runs on every push/PR) and `release.yml` (runs only on a `v*.*.*` tag push — re-tests as a gate, builds+pushes the Docker image to `ghcr.io`, creates a GitHub Release with autogenerated notes).
- Keep both workflows minimal. No build matrix, no extra job splitting, unless a concrete need arises later (e.g. multi-arch images).
- Set these up early — ideally right after the initial schema/skeleton commit, not bolted on at the end — so every meaningful commit from the start is covered by CI.
- Test suite starts thin by design (see `DESIGN.md` §3 for the floor: settings cache, engine registry hot-swap, one happy-path API test per route group, schema application test). Expand over time; don't block the first alpha tag on exhaustive coverage.

**Platform distribution — Docker first, native installers later, Windows before macOS/Linux.** The Docker image (pushed to `ghcr.io`) is the only release artifact for now and already runs cross-platform via `docker-compose up`. Native, no-Docker installers (double-click apps) are a distinct future phase — Windows first, then macOS, then Linux — that starts only when explicitly requested. **Do not scaffold any installer packaging, build scripts, or CI jobs for native installers until that request happens.** Until then, keep `app/` itself platform-agnostic (no OS-specific code paths) so that whenever installer work starts, it's a packaging/build concern layered on top of the existing app, not a rewrite. Full detail in `DESIGN.md` §3 ("Platform distribution roadmap").

## Schema

Full schema in `DESIGN.md` §4. Tables: `identities`, `face_embeddings`, `source_images`, `detections`, `models`, `settings`. Read the design doc's rationale for each before modifying — several columns exist specifically to support features described below (e.g. `face_embeddings.model_id` exists so switching active face models doesn't invalidate enrollment history).

Schema file should live at `app/db/schema.sql` and be the single source of truth — apply it idempotently on startup (`CREATE TABLE IF NOT EXISTS`), seed `models` and `settings` tables only if empty.

## API contract

Full endpoint list in `DESIGN.md` §5. Key things to preserve:

- All `/api/*` routes require `X-API-Key` header. Page routes (`/`, `/identities/{id}`, etc.) are not API-key gated (LAN-trusted assumption).
- `PUT /api/detections/{id}/label` is intentionally shared between faces and objects, and between the casual-correction UI surfaces (galleries, test page, tag page) and is a simpler contract than the formal `/api/review/*` flow. Do not create separate endpoints per surface — every "fix this label" UI action should call this one endpoint.
- `/api/review/*` (confirm/reject/reassign/bulk) is faces-focused and includes ranked `suggested_matches` — this is the deliberate triage workflow, distinct from casual correction.
- Cursor pagination pattern (`?cursor=<detected_at>&limit=N` → `{items, next_cursor, has_more}`) is used for both galleries and the review queue — keep it consistent.

## Key behavioral requirements

1. **Hot-swap models, no restart.** `PUT /api/models/{id}/activate` must reload the in-process engine (unload old weights, load new) live. Needs a lock around the active-engine reference so in-flight detection calls don't read a half-swapped state. See `core/engine_registry.py` in the target layout.
2. **Settings are live, not requiring restart.** Engines read from an in-process settings cache, refreshed whenever `PUT /api/settings/*` is called.
3. **Face embeddings tagged by model.** Matching only compares against embeddings generated by the currently active face model (`face_embeddings.model_id`). Multiple model generations can coexist in the DB.
4. **Justified gallery + infinite scroll**, per identity (person or object class). Backed by cursor pagination on `detections`. Thumbnails are pre-saved crops (`crop_path`), not re-cropped at render time. Implement the justified layout packing in vanilla JS — do not pull in a gallery library, per the minimal-deps value.
   - **Cropping happens at detect-time, not display-time.** Every detection row gets its own tight crop on just that face/object's bbox (padded by `system.crop_padding`, clamped to image bounds), saved immediately to `crop_path`. A group photo with five faces produces five separate crops. A person's/class's gallery only ever shows their own face/object crops — never the full source image, no matter how large or cluttered the original photo was. The full source frame is only surfaced elsewhere (e.g. the tag page) via `source_image_id`.
5. **Multi-face tagging per image** — hover a face bbox on `/tag/{source_image_id}` to see its label, click to assign/correct. Bbox overlays are absolutely-positioned divs computed from stored bbox coordinates scaled to the displayed image size.
6. **Review queue is faces-only for now.** Objects get the simple inline-correct affordance, no dedicated queue, no ranked suggestions.
7. **Detection and enrollment accept file upload, URL, or base64 — exactly one.** Every `/api/detect/*` and `/api/*/enroll` endpoint takes one of three: a `file` multipart upload, an `image_url` field, or an `image_base64` field (raw base64 or a `data:image/...;base64,` URI — strip the prefix before decoding). Zero or more than one of the three is a 400. URL fetching deliberately allows private/local IP ranges (192.168.x.x, 10.x.x.x, localhost) — this is a LAN tool meant to pull images from things like a local camera or Pi-hosted service, so **do not add SSRF/private-IP blocking**. Fetch timeout and max size are NOT hardcoded — they're `system.url_fetch_timeout_seconds` (default 10) and `system.url_fetch_max_size_mb` (default 25) in the `settings` table, read from the same live settings cache as every other threshold, editable via the same `/api/settings/*` endpoints and settings page UI — no special-casing required, the settings page is driven generically off the table. Do add Content-Type/content-sniffing validation before decode regardless of source. All three input paths converge to the same `image_bytes` → Pillow `Image.open` step immediately after acquisition — no downstream branching on input source. Supported formats: JPEG, PNG, WEBP, BMP, GIF, TIFF, HEIC/HEIF — validate by sniffing actual content (Pillow format detection), never by trusting file extension or declared Content-Type alone.
8. **GPU is a runtime toggle, not a build-time decision.** `system.use_gpu` (default `false`) selects `CUDAExecutionProvider` vs `CPUExecutionProvider` for onnxruntime at engine-load time. Toggling it goes through the same hot-swap lock as model switching — no restart. If GPU is requested but unavailable (no card, drivers, or passthrough not configured), fall back to CPU automatically and log it clearly — never fail the request outright. The actual active provider (CPU or GPU) must be visible via `GET /api/health` so it's never silently wrong. Docker GPU passthrough on Windows requires the Docker Desktop WSL2 backend specifically — do not assume this is configured; if `onnxruntime` and `onnxruntime-gpu` turn out to conflict at install time, flag that tradeoff explicitly rather than silently resolving it (see `DESIGN.md` §2 "GPU support" for the fallback approach).

## Open decisions flagged for you

These are called out in `DESIGN.md` §7 — resolve sensibly, but flag the choice made in your PR/commit description rather than silently picking one:

- GPU passthrough status on this specific Docker Desktop install (the `system.use_gpu` setting and fallback logic are designed; whether the WSL2/CUDA prerequisite is actually configured on this machine still needs confirming when this gets implemented)
- Whether `onnxruntime` and `onnxruntime-gpu` can coexist in the same install without conflict — if not, resolve via build arg (see `DESIGN.md` §2) and flag it
- Exact async mechanism for model downloads (BackgroundTasks vs. simple asyncio task)
- Whether model activation should be synchronous (blocks until swapped) or return a job id, based on observed weight-load time
- Crop padding application details (`system.crop_padding`, default 0.2, expand bbox before crop, clamp to image bounds)
- Container registry for release images (default: `ghcr.io`, see `DESIGN.md` §7 item 7)

## Suggested build order

1. Repo skeleton: `LICENSE` (MIT), `app/__init__.py` with `__version__ = "0.1.0"`, `app/__main__.py` (CLI entry point, see Run modes above), directory layout per `DESIGN.md` §8, `requirements.txt`, `pyproject.toml` (ruff config)
2. `.github/workflows/test.yml` (lint + pytest, even with near-zero tests at this point — establish the gate immediately) and `.github/workflows/release.yml` (tag-triggered build/push/release)
3. Schema + `db/store.py` raw sqlite access layer, seed data for `models`/`settings`; first real tests (schema application test)
4. `core/settings_cache.py`, `core/engine_registry.py` skeletons (no real models loaded yet); tests for settings cache get/set/reset
5. Face engine (InsightFace) + object engine (YOLO) wrappers, behind the registry; tests for hot-swap lock behavior
6. `/api/detect/*` end to end (upload → detect → write `detections`/`source_images` rows → return JSON); happy-path API test
7. `/api/identities/*`, `/api/faces/enroll`, galleries
8. Models management endpoints + hot-swap
9. Settings endpoints
10. Review queue + casual correction endpoint
11. Multi-face tagging endpoint + page
12. Jinja pages and static JS/CSS last, once the API is solid — dashboard, gallery (justified + infinite scroll), test page, review page, tag page, models page, settings page
13. Dockerfile + docker-compose, volumes for `models/` and `data/`
14. Verify both run modes work identically: `python -m app` locally AND `docker compose up` — same routes, same behavior, before tagging the first release
15. Tag `v0.1.0-alpha.1` once the above is functional end to end — first release workflow run
